# Contributing to SensorHub Cloud Documentation
Thank you for contributing to the SensorHub Cloud documentation sample.

This repository is a fictional docs-as-code portfolio project. It demonstrates technical writing, information architecture, Markdown authoring, API documentation, troubleshooting content, release notes, and documentation review workflows.

Although this is a portfolio project, contributions should follow the same standards used in a professional documentation environment.

## Documentation goals
The SensorHub Cloud documentation should help users:
- Understand what SensorHub Cloud does.
- Set up and configure the product successfully.
- Complete common Administrator and user tasks.
- Troubleshoot common issues.
- Use the SensorHub API with clear examples.
- Understand product changes through release notes.

Every documentation change should improve the content's accuracy, usefulness, consistency, or ease of finding.

## Audience
Write for these audiences:
- **Administrators** who configure users, roles, gateways, alerts, and account settings
- **Support teams** who troubleshoot device, gateway, and data issues.
- **Developers** who use the SensorHub API to retrieve sensor data.
- **New users** who need a clear path to get started.

Before writing, identify which audience the article supports.

## Content types
This repo includes the following documentation types:
- Overview articles
- Quickstarts
- How-to procedures
- Troubleshooting articles
- API documentation
- Error code references
- Release notes
- Style guide content

Use the correct format for the type of content you are creating.

## Writing standards
Use clear, direct language.
### Do
- Write short paragraphs.
- Use active voice.
- Start procedures with an active verb.
- Break complex tasks into numbered steps.
- Define technical terms before using them heavily.
- Include expected results when helpful.
- Use examples for API requests, responses, errors, and configuration values.
- Write headings that describe the user task or topic.

### Avoid
- Long introductory paragraphs.
- Marketing language.
- Unexplained acronyms
- Vague phrases such as "simply," "easily," or "just."
- Mixing multiple tasks into one procedure.
- Assuming the user already understands the product.

## Review checklist
Before submitting a change, confirm that the content:
- Has a clear audience.
- Solves a real user problem.
- Uses the correct article type.
- Follows the folder and file naming standards.
- Uses active voice.
- Includes prerequisites when needed.
- Includes verification steps for procedures.
- Includes realistic examples for API documentation.
- Does not include proprietary information.
- Does not include real company names, customer names, credentials, URLs, or internal system details.
- Uses consistent terminology.
- Has been checked for spelling, grammar, and formatting.

## Confidentiality requirements
This repository is a fictional portfolio project. Do not add:
- Proprietary documentation from a current or former employer.
- Internal screenshots.
- Customer data.
- Real API keys or tokens.
- Private endpoints.
- Internal architecture diagrams.
- Confidential product names.
- Non-public business processes.

All examples would be fictional, anonymized, or created specifically for this project.

## Questions
For portfolio review purposes, open an issue or add comments in the pull request explaining the documentation decision you want reviewed.
